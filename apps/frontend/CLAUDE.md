# Frontend App — Claude Guidelines

> Applies to any React-based frontend in this monorepo — Expo / React Native and Next.js (App Router).
> Where behaviour differs between frameworks, both are documented explicitly.

---

## Folder Structure

```
app/                  ← routing entry points (Expo Router screens OR Next.js App Router pages)
components/
  ui/                 ← pure presentational, no data fetching, fully reusable
  features/           ← domain-specific, may fetch data
services/             ← all API integration logic (no fetch/axios in components)
hooks/                ← shared custom hooks
stores/               ← Zustand global UI state
lib/                  ← utility functions, constants, helpers
```

- `ui/` components must never import from `features/`.
- Never call `fetch` or the generated API client directly inside a component — always go through `services/`.

**Framework notes:**
- **Expo Router** — screens live in `app/`, file-based routing, each file is a screen.
- **Next.js App Router** — pages live in `app/`, each `page.tsx` is a route. Server Components fetch data at the route level; pass results down as props to client components.

---

## State Management

| State type | Tool |
|---|---|
| Server state (fetch, cache, sync) | TanStack React Query |
| Global UI state (modals, sheets, drawers, preferences) | Zustand |
| Local component state | `useState` / `useReducer` |

- Do not mix server state and UI state in the same store.
- Avoid prop-drilling beyond 2 levels — lift to Zustand instead.

**Next.js note:** TanStack React Query runs in Client Components only (`"use client"`). Server Components fetch directly using the generated API client — do not wrap server fetches in `useQuery`. Use React Query only where interactivity or caching on the client is needed.

---

## API Integration

- All API calls live in `services/` — one file per domain (e.g. `services/auth/service.ts`).
- Use **TanStack React Query** for all client-side queries and mutations.
- The API client in `packages/swag/src/api.ts` is generated by Orval — do not hand-write API types.
- After any backend contract change: regenerate with `pnpm codegen` before writing frontend code against the new shape.

**Next.js note:** Server Component data fetching can use the generated client directly (no React Query). Add `cache` and `revalidate` options on the `fetch` calls inside the generated client if needed for ISR/SSR caching.

## Defensive Data Handling

Every component that fetches data must handle all three states — never assume data is present:

```tsx
const { data, isLoading, isError } = useQuery(...)

if (isLoading) return <Skeleton />
if (isError)   return <ErrorView />
// only now is data safe to render
```

- Use optional chaining everywhere: `user?.profile?.avatar` — never assume nested properties exist.
- Never render `undefined` directly.

**React Error Boundaries are required at the route/page level.** One uncaught render error crashes the entire UI without them.

```tsx
<ErrorBoundary fallback={<ErrorPage />}>
  <FeatureSection />
</ErrorBoundary>
```

| Framework | Error Boundary approach |
|---|---|
| Expo Router | Wrap each screen's root in `<ErrorBoundary>` |
| Next.js App Router | Use `error.tsx` files — they are the built-in Error Boundary for each route segment |

## Form Validation

- Use **react-hook-form** + **Zod** via `@hookform/resolvers/zod`.
- Define schema once — it is both the validation rule and the TypeScript type.
- Shared schemas (reused in the backend) go in `packages/schemas/`.

```ts
const schema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
})
type FormInput = z.infer<typeof schema>
```

- Validate before calling any API mutation.
- Never call an API with unvalidated user input.

**Next.js note:** For Server Actions, validate the incoming `FormData` with the same Zod schema on the server side before processing. Do not trust client-side validation alone.

## Error Handling (API errors)

The backend returns a consistent error shape — always switch on `errorCode`, never parse `message`:

```ts
// services/auth/service.ts
try {
  await loginMutation.mutateAsync(input)
} catch (err) {
  if (err.errorCode === 'VALIDATION_ERROR') { ... }
  if (err.errorCode === 'UNAUTHORIZED')     { ... }
  if (err.errorCode === 'CONFLICT')         { ... }
}
```

`message` is for display only and may change. `errorCode` is the contract.

## Rendering Strategy (Next.js only)

Choose the right rendering strategy per route — do not default everything to client-side:

| Route type | Strategy |
|---|---|
| Public, mostly static content | Static (`generateStaticParams` / `force-static`) |
| Data changes periodically | ISR (`revalidate: N`) |
| Data is user-specific or real-time | SSR (`force-dynamic`) or Client Component with React Query |
| Highly interactive UI (forms, live updates) | Client Component (`"use client"`) |

- Prefer Server Components by default — add `"use client"` only when interactivity is required.
- Never put secrets or server-only logic inside a Client Component — they are shipped to the browser.

---

## TypeScript

- `strict: true` is enabled — do not disable it.
- No `@ts-ignore` without a comment explaining why.
- No `any` — use `unknown` and narrow explicitly.

---

## Component Rules

- Keep components small — if a component exceeds ~150 lines, split it.
- No business logic in components — extract to hooks or services.
- No direct store reads deep in `ui/` components — pass props down or use a feature wrapper.
- `ui/` components must be framework-agnostic where possible — avoid Expo-specific or Next.js-specific imports inside `ui/`.
