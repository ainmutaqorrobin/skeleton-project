# API — Claude Guidelines

> NestJS backend. Follow these rules in every file you touch in this app.

---

## Folder Structure

```
src/
  modules/            ← one folder per domain (e.g. users/, auth/, orders/)
    *.module.ts
    *.controller.ts
    *.service.ts
    *.dto.ts          ← request/response shapes with class-validator decorators
  common/
    filters/          ← GlobalExceptionFilter
    pipes/            ← validation pipe setup
    interceptors/     ← request ID, logging
    decorators/       ← custom decorators (e.g. @CurrentUser)
  db/
    migrations/       ← Kysely migration files
    seed/             ← seeder scripts
    types/            ← Kysely codegen output (do not hand-edit)
  config/             ← env validation schema, config service setup
main.ts
```

---

## Bootstrap Checklist

`main.ts` must always register these before `app.listen()`:

```ts
// 1. Global validation pipe
app.useGlobalPipes(new ValidationPipe({ whitelist: true, forbidNonWhitelisted: true }))

// 2. Global exception filter
app.useGlobalFilters(new GlobalExceptionFilter())

// 3. Swagger + spec file sync
const document = SwaggerModule.createDocument(app, config)
writeFileSync("./swagger-spec.json", JSON.stringify(document))
SwaggerModule.setup("api", app, document)
app.use("/swag", apiReference({ spec: { content: document } }))
```

Never remove or comment out any of these — they are required for correct runtime behaviour.

---

## Request Validation

- Every controller input must have a DTO class with `class-validator` decorators.
- `whitelist: true` strips unknown fields automatically — do not manually strip them.
- `forbidNonWhitelisted: true` rejects requests with unexpected fields — do not bypass this.
- Never access `req.body` directly — always use typed DTOs.

---

## Standardized Error Responses

**Every error must return this exact shape — no exceptions:**

```json
{
  "statusCode": 400,
  "errorCode": "VALIDATION_ERROR",
  "message": "One or more fields are invalid.",
  "details": [{ "field": "email", "message": "Invalid email format" }],
  "timestamp": "2024-01-01T00:00:00.000Z",
  "path": "/api/users",
  "requestId": "uuid"
}
```

- `errorCode` is a machine-readable string from the shared `ErrorCode` enum in `packages/schemas/`.
- `GlobalExceptionFilter` handles all error sources — do not return raw errors from controllers.
- Never expose stack traces, SQL queries, or file paths in responses.
- In production, `INTERNAL_ERROR` message must always be `"An unexpected error occurred."`.

**Error source mapping (handled inside `GlobalExceptionFilter`):**

| Source | Maps to |
|---|---|
| `HttpException` | use its status + map to `ErrorCode` |
| `ValidationPipe` error | `400 VALIDATION_ERROR` + populate `details` |
| Postgres unique constraint (`23505`) | `409 CONFLICT` |
| Postgres foreign key violation (`23503`) | `400 BAD_REQUEST` |
| `better-auth` errors | `401 UNAUTHORIZED` or `403 FORBIDDEN` |
| Anything unrecognized | `500 INTERNAL_ERROR` — log full stack, return no internals |

## Environment Variables

- All env vars are validated at startup in `config/env.validation.ts` using Joi or Zod.
- A missing required var must crash the app at boot — not at the first usage.
- Never read `process.env.SOMETHING` directly in modules — inject via `ConfigService`.

---

## Database (Kysely)

- Use **Kysely** for all queries — no raw SQL strings unless absolutely necessary.
- Never hand-edit `db/types/` — it is generated by `pnpm db:codegen` from the live schema.
- Run `pnpm db:codegen` after every migration before writing queries against new columns.
- Kysely codegen requires a running DB — ensure DB is up before running it.

---

## Migrations

- File naming: `YYYYMMDDHHMMSS_short_description.ts`
- Every migration file must export both `up` and `down`.
- **Never edit a migration file after it has been merged to `main`.** Write a new one instead.
- **Never drop a column in the same PR that removes it from code.** Two-step: remove from code first, drop column in follow-up migration.
- Always rebase from `main` before creating a new migration to avoid timestamp/ordering conflicts.
- Run migrations via: `pnpm db:migrate`

---

## API Documentation

- Decorate **all** DTO properties with `@ApiProperty()` — undocumented fields are invisible to Orval and will not appear in frontend types.
- Decorate all controller methods with `@ApiResponse()` for documented response shapes.
- `swagger-spec.json` is auto-written on every app start — commit it before merging to `main`.

---

## Authentication

- Use **`better-auth`** — do not roll custom auth logic.
- Protect routes with the auth guard provided by `better-auth` integration.
- Never store raw passwords — `better-auth` handles hashing.
- Pin the `better-auth` version — it is a newer library and may have breaking changes.

---

## Logging

- Use structured JSON logging via **Pino** — no `console.log` in production code.
- Every log entry must include: `timestamp`, `level`, `requestId`, `userId` (if authenticated), `message`.
- Never log sensitive fields: passwords, tokens, card numbers. Redact them.
- The `GlobalExceptionFilter` logs every error server-side with full stack trace.

---

## Rate Limiting

- `@nestjs/throttler` is registered globally.
- Auth endpoints (`/auth/login`, `/auth/register`) must have stricter limits than general routes.
- Do not remove throttler from any endpoint unless there is a documented reason.

---

## Seeder

- Seeder lives in `db/seed/`.
- All seed operations must be **idempotent** — use upsert, never bare insert.
- Run via: `pnpm db:seed`
- Cover realistic edge cases, not just happy-path rows.

---

## TypeScript

- `strict: true` is enabled — do not disable it.
- No `@ts-ignore` without a comment explaining why.
- No `any` — use `unknown` and narrow explicitly.
- Return types on all service methods must be explicit — do not rely on inference for public API.
